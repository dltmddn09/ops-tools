# 빌드 브랜치 타임라인 — 스펙

머지 내역 확인 시 특정 커밋이 어느 브랜치에 포함됐는지 확인할 수 있도록,
MAINLINE / BETA / ALPHA 브랜치 흐름을 한눈에 보는 타임라인. 주 1회 갱신.

기존 xlsx 생성 방식(`CD 브랜치 현황 관리/build_timeline.py`)을 대체하는 HTML 버전.
**이벤트(브랜치/패치)만 입력하면 레인별 버전 구간과 색 전환 시점은 규칙 엔진이 자동 파생**한다.

## 파일 구성

| 파일 | 역할 | 갱신 빈도 |
|------|------|-----------|
| `index.html` | 뷰어 + 편집기 마크업/스타일 | 거의 없음 |
| `app.js` | 규칙 엔진 + 렌더러 + 편집기 로직 | 거의 없음 |
| `data.js` | 타임라인 데이터 (`window.TIMELINE_DATA`) | **매주** |

## 브랜치 운영 규칙

- 레인 구성: **MAINLINE → BETA → ALPHA** (MAINLINE이 상위)
- 패치는 항상 **ALPHA에서만** 발생 (`patchLaneId`)
- 버전 형태: `AA.BB.CC` (CC = 핫픽스, BB = 정기 패치, AA = 대형 업데이트)
- **격주로 두 가지 패치 방식이 번갈아 진행**
  - **정규 브랜치 주차**(`mode: "regular"`): MAINLINE → BETA → ALPHA 순으로 브랜치 후 ALPHA에서 패치
  - **ALPHA 단독 주차**(`mode: "solo"`): 별도 브랜치 없이 ALPHA에 추가 데이터만 받아서 패치
- 같은 날 핫픽스가 추가로 발생할 수 있음 (자동으로 패치 2번째 행에 표시됨)

## 색 전환 규칙 (app.js `deriveSegments`가 자동 적용)

| 상황 | 전환 시점 |
|------|-----------|
| 브랜치를 **받는** 레인 (`to`) | 브랜치 **당일**부터 해당 버전 |
| 브랜치를 **보낸** 레인 (`from`) | `sourceNext` 지정 시 **다음 날**부터 그 버전 |
| 단독(solo) 주차 버전 | **직전 패치/핫픽스 다음 날**부터 |
| 핫픽스 | **당일**부터. 색은 "가장 가까운 직전 빌드"(CC-1, 없으면 CC-2 …)색에서 한 단계 더 진하게 자동 배정 — 핫픽스가 연달아 나도 매번 색이 달라짐 |
| 정규(regular) 주차 버전 | 패치 이벤트 자체는 색 전환을 만들지 않음 — **브랜치 이벤트가 실제 입력되기 전까지 이전 버전 색 유지** |

> 마지막 규칙이 xlsx 시절 매번 손으로 재검토하던 "ALPHA는 BETA→ALPHA 브랜치 전까지
> 이전 버전 색 유지" 규칙이다. 이제 엔진이 보장하므로 별도 검토 불필요.

### 브랜치 버전 ↔ 보낸 레인 버전 자동 동기화 (`syncSourceVersion`)

브랜치 이벤트의 버전은 "보낸 레인이 그 날 실제로 갖고 있던 버전"과 항상 일치해야 한다
(그렇지 않으면 "1.14.00에서 1.13.00을 브랜치했다" 같은 모순이 생김). 그래서:

- 브랜치를 **생성/수정**할 때마다 `syncSourceVersion`이 보낸 레인(`from`)의 그 시점 버전을
  결정하던 설정(직전 브랜치의 `sourceNext`, 또는 `initialVersions`)을 **소급 보정**한다.
- 보낸 레인이 다시 다른 브랜치로 받은 버전이면(중간 레인), 그 브랜치까지 **재귀적으로 연쇄 보정**된다
  (예: BETA→ALPHA 버전을 고치면 MAINLINE→BETA의 `sourceNext`까지 올라가서 고쳐짐).
- `render()`가 호출될 때마다 **전체 브랜치에 대해 자동 치유(`healBranchVersions`)**를 돌리므로,
  과거에 수동 편집 등으로 어긋난 data.js를 열기만 해도 정합 상태로 복구된다.
- 그래도 남는 불일치(예: 순서가 꼬여 자동 보정이 못 미치는 경우)는
  `deriveSegments`가 "브랜치 버전 불일치" 경고로 표시한다.

### 버전 이름 전역 변경 (`renameVersion`)

레인 구간을 클릭해 버전명을 바꾸면, 그 버전을 참조하는 모든 곳
(`branches[].version/.sourceNext`, `patches[].version`, `initialVersions`, `versionColors`)이
한 번에 함께 바뀐다. 버전은 빌드의 정체성이므로 반쪽만 바뀌는 상태를 만들지 않기 위함.

## 데이터 스키마 (`data.js`)

```js
window.TIMELINE_DATA = {
  title: "…",                 // 상단 타이틀
  startDate: "YYYY-MM-DD",    // 표시 범위
  endDate:   "YYYY-MM-DD",
  lanes: [                    // 위→아래 순서, 인접 레인 사이에 브릿지 행 자동 생성
    { id: "mainline", name: "MAINLINE", labelBg: "#1F4E79" }, …
  ],
  patchLaneId: "alpha",       // 패치가 발생하는 레인
  initialVersions: { laneId: "버전" },  // startDate 시점의 레인별 버전
  branches: [                 // 브랜치 이벤트
    { date, time, from, to, version, sourceNext? }, …
  ],
  patches: [                  // 패치/핫픽스 이벤트
    { date, version, type: "patch"|"hotfix", mode?: "regular"|"solo" }, …
  ],
  versionColors: { "버전": "#hex" }  // 색 오버라이드(없으면 자동 배정)
};
```

## 표시 규칙 (xlsx 레이아웃 계승)

- 가로축 = 날짜(1일 1컬럼), 요일/날짜 2단 헤더, 첫 컬럼(레인 레이블) sticky 고정
- 같은 레인의 같은 버전 구간은 colspan 병합, 버전명 1개만 표시
- 패치가 있었던 날의 헤더는 빨강, 주말은 회보라, 주 단위로 헤더 명암 교차
- 브릿지 행: 브랜치 발생일에만 해당 버전 색 + **시간만 표기** ("브랜치" 텍스트 없음)
- 패치 행: `버전 패치` / `버전 핫픽스` 빨간 글씨, 같은 날 2건이면 2행
- 오늘 날짜 강조 / 하단 범례는 사용하지 않음
- 셀 호버 시 툴팁 (레인·버전·구간 / 브랜치 상세 / 패치 주차 방식)

## 캘린더 직접 편집 (⚙ 편집 모드에서만 활성화)

브랜치는 항상 위→아래 레인으로만 발생하므로, 편집 메뉴를 거치지 않고
**캘린더 셀을 직접 클릭**해서 그 흐름 그대로 입력할 수 있다.

| 클릭 대상 | 동작 |
|-----------|------|
| 레인 사이 빈 칸 (브릿지 행) | 브랜치 생성 팝오버. 방향은 위→아래로 고정, 버전은 그 날짜의 위 레인 버전을 자동 채움(최상위 레인이면 "다음 버전"도 +0.02로 제안) |
| 기존 브릿지 셀 | 해당 브랜치 수정/삭제 팝오버 |
| 패치 행 빈 칸 | 인라인 텍스트 입력. **"패치"/"핫픽스"라는 단어만 쳐도** 종류와 배경·폰트색이 자동 적용되고, 버전을 안 쓰면 그 시점 패치 레인 버전(핫픽스는 CC+1)으로 자동 추론. 버전을 직접 써도 됨(예: `1.13.00 패치`) |
| 기존 패치 셀 | 해당 패치/핫픽스 수정/삭제 팝오버 |
| 레인의 색칠된 구간 | 버전명·색 직접 수정 팝오버 (버전을 바꾸면 `renameVersion`으로 전체 이벤트에 반영) |

브랜치 팝오버에서 버전을 정하면 `syncSourceVersion`이 보낸 레인 쪽도 자동으로 맞춘다(위 "색 전환 규칙" 참고).
패치 레인으로 들어오는 브랜치를 생성하면, 같은 버전으로 등록된 `mode: "solo"` 패치는 자동으로 `"regular"`로 전환된다.

## 편집 패널

- 레인 / 브랜치 이벤트 / 패치·핫픽스 각 항목 행에 **해당 버전(레인) 색 스트라이프 + 옅은 배경**이 붙어
  캘린더와 시각적으로 연결된다.
- 브랜치·패치 목록은 **최신순 정렬**. 현재 표시 기간(`startDate` 이전) 이벤트는
  "지난 이벤트 N건" 아코디언으로 접혀 목록이 길어져도 스크롤이 늘어나지 않는다.
- **표시 기간 프리셋** — 최근 4주 / 최근 8주 / 전체 버튼으로 `startDate`/`endDate`를 한 번에 조정.
- 페이지를 열면 캘린더가 자동으로 가장 최근 주로 스크롤된다.

## 갱신 워크플로우

1. 페이지(GitHub Pages)에서 **⚙ 편집** → 캘린더에서 직접 클릭하거나 우측 패널에서 편집
   - 편집 내용은 브라우저 localStorage에 자동 임시 저장됨(새로고침해도 안 날아감)
   - 다음에 열었을 때 임시 저장본이 있으면 상단에 "불러오기 / 삭제" 배너가 뜸
2. **⬇ data.js 다운로드** → 받은 파일 경로를 Claude에게 알려주고 "반영해줘"
   - 또는 일정을 텍스트로 알려주면 Claude가 data.js를 직접 수정
3. Claude가 레포의 `data.js`를 교체 → `?selftest=1`로 회귀 검증 → commit & push
4. GitHub Pages 자동 반영(추가 대기 없이 보통 1분 내) → 팀원들은 같은 URL에서 최신 타임라인 확인
5. 다운로드했던 로컬 data.js 파일은 반영 후 더 이상 필요 없음 (요청 시 삭제)

> 다운로드 후 브라우저만 닫고 Claude에게 전달하지 않으면, 편집 내용은 본인 브라우저에만
> 남아 있고 팀원들에게는 반영되지 않는다 — 반드시 push까지 가야 공유 완료.

## 검증

`index.html?selftest=1` 로 열면 규칙 엔진(구간 파생, 버전 증가 헬퍼, 소급 보정,
버전 이름 변경, 핫픽스 색 구분)을 **고정 픽스처**(라이브 data.js와 무관) 기준으로 검증한
PASS/FAIL 목록이 페이지 상단에 표시된다. data.js를 갱신해도 이 셀프테스트 결과는 영향받지 않는다.
`?edit=1` 로 열면 편집 패널이 열린 상태로 시작하고, `?demo=bridge` / `?demo=patch` 로
브랜치/패치 팝오버를 즉시 띄워 스크린샷 확인이 가능하다.

data.js를 갱신할 때마다 최소 다음을 확인한다:
1. `?selftest=1` — ALL PASS
2. 편집 패널 상단에 "⚠ 확인 필요" 경고가 없는지 (브랜치 버전 불일치 등)
3. 새로 추가된 구간이 의도한 색으로 옆 레인과 구분되는지 (특히 자동 배정색이 인접 버전과 비슷한 계열로 겹치는 경우가 있어 육안 확인 필요)
