# 빌드 브랜치 타임라인 작업 규칙

이 폴더에서 작업할 때는 먼저 [SPEC.md](SPEC.md)를 읽고 시작할 것. 브랜치 운영 규칙, 색 전환 규칙, 데이터 스키마, 갱신 워크플로우가 모두 거기 정리되어 있음.

## 핵심 원칙

- 매주 갱신되는 파일은 `data.js` 하나뿐. `index.html`/`app.js`는 거의 안 바뀜.
- 레인 구성은 **MAINLINE → BETA → ALPHA** 고정, 패치는 항상 **ALPHA에서만** 발생.
- 버전 구간/색 전환 시점은 손으로 계산하지 않는다 — `app.js`의 규칙 엔진(`deriveSegments`, `syncSourceVersion`, `healBranchVersions`)이 자동 파생/자동 치유한다. data.js에 이벤트(브랜치/패치)만 넣으면 됨.

## data.js 갱신 시 반드시 확인할 것

1. `index.html?selftest=1` 로 열어서 **ALL PASS** 확인
2. 편집 패널 상단에 "⚠ 확인 필요" 경고(브랜치 버전 불일치 등)가 없는지
3. 새로 추가된 구간이 인접 레인과 색이 겹치지 않는지 육안 확인 (자동 배정색이 인접 버전과 비슷한 계열로 겹치는 경우가 있음)

## 갱신 워크플로우 — push까지 가야 팀에 반영됨

사용자가 GitHub Pages에서 편집 → **⬇ data.js 다운로드** → 경로를 전달하면, 레포의 `data.js`를 교체하고 `?selftest=1` 회귀 검증 후 commit & push. **다운로드만 하고 전달받은 파일을 반영·push하지 않으면 사용자 브라우저에만 남아있고 팀원에게는 공유되지 않는다.**
